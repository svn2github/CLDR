GMT Translations

2003-07-24, MD

Background

When ICU formats or parses a date pattern containing a "z" parameter (for time-zone), then it uses the Olson IDs to look up a localized name. For example:

"hh 'o''clock' a, zzzz" ->> 12 o'clock PM, Pacific Daylight Time

If there are no localized names for a given Olson ID, then it falls back to using a format like "GMT+12:00", so for another time-zone the above pattern might produce:

"hh 'o''clock' a, zzzz" ->> 11 o'clock AM, GMT-09:00

For more information, see:

  • http://oss.software.ibm.com/icu4j/doc/com/ibm/icu/util/TimeZone.html
  • http://oss.software.ibm.com/icu4j/doc/com/ibm/icu/text/SimpleDateFormat.html

Problem

Currently that fallback format is hard-coded, which is prima facie a problem, since one can't translate "GMT" or ":" into strings more appropriate for other languages, and the ordering of elements is fixed. Moreover, the GMT+/- format itself may not be customary in a given country. If the time-zone being reported is in India, but the localized time is for a Serbian, for example, it would be much more useful to note the offset from Serbian time rather than the offset from GMT (actually UTC).

For examples of current zone string localizations, see:

  • http://oss.software.ibm.com/cvs/icu/~checkout~/icu/source/data/locales/en.txt

Proposed Solutions

Add a ZoneFallback resource, containing a base Olson ID, and some fallback strings, all of which can be localized for each locale. Each string is itself a date format. The base time is an ID used for normalization. Examples (just for illustration):

Root:

Zone_Base: "GMT"
Positive_Zone: "'GMT+ hh:mm"
Negative_Zone: "'GMT-'hh:mm"
Abb_Positive_Zone: "'GMT+ 'hh:mm"
Abb_Negative_Zone: "'GMT-'hh:mm"

Open Issue: should we use UTC or GMT in the Root? Probably better to use GMT for compatibility.

en_US:

Zone_Base: "America/New_York"
Positive_Zone: "'Eastern Time+ 'hh:mm"
Negative_Zone: "'Eastern Time-'hh:mm"
Short_Positive_Zone: "'ET+ 'hh:mm"
Short_Negative_Zone: "'ET-'hh:mm"

de_DE:

Zone_Base: "Europe/Paris"
Positive_GMT: "'Mitteleuropäische Zeit+ 'hh.mm"
Negative_GMT: "'Mitteleuropäische Zeit-'hh.mm"
Short_Positive_GMT: "'MEZ+ 'hh.mm"
Short_Negative_GMT: "'MEZ-'hh.mm"

If there is no specific name for a zone ID, then these values will be used to compose a fallback. The Zone_Base will be used to find the difference in hours/minutes. Those values will then be formatted with the available localization strings, according to whether the difference is positive or negative, and weather the long or short "z" parameter is used in the format string.

For parsing, all of the localization strings will be attempted, plus the ones for the Root. In addition, we will be somewhat more lenient by permitting plus-signs and punctuation between hh and mm to be omitted.

We would generate initial (static) data for the above having a tool pass over our data and construct some defaults. E.g. we would get the punctuation for the hh:mm string from the existing short time strings, and have the base be GMT. We would then pass this out to the translation sites for better data.
