CLDR Test Data

MD, 2004-07-15

Very Rough Draft

This set of files provides the ability for people to check the results of their implementations against the data provided in CLDR. The format has not yet been approved by the CLDR technical committee, so this is just a working draft; feedback is welcome.

Each file contains number of test areas, indicated by a particular element. Currently these are for number, date, and collation. Within each test area, there are a number of elements called 'results'. Each of those lines contains zero or more settings (given by attributes) and a result. The settings are additive: each setting only needs to be provided if it is different than what is in the previous result. Thus, for example,

  <result dateType="none" input="1900-01-31T00:00:00Z" timeType="short">00:00</result>
  <result timeType="medium">00:00:00</result>
  <result timeType="long">00:00:00 GMT</result>
  <result timeType="full">00:00:00 GMT</result>
  <result dateType="short" timeType="none">31-01-00</result>
  <result timeType="short">31-01-00 00:00</result>

is equivalent to:

  <result dateType="none" input="1900-01-31T00:00:00Z" timeType="short">00:00</result>
  <result dateType="none" input="1900-01-31T00:00:00Z" timeType="medium">00:00:00</result>
  <result dateType="none" input="1900-01-31T00:00:00Z" timeType="long">00:00:00 GMT</result>
  <result dateType="none" input="1900-01-31T00:00:00Z" timeType="full">00:00:00 GMT</result>
  <result dateType="short" input="1900-01-31T00:00:00Z" timeType="none">31-01-00</result>
  <result dateType="short" input="1900-01-31T00:00:00Z" timeType="short">31-01-00 00:00</result>

Date

For dates, the input value indicates what is to be formatted. It is represented in ISO 8601 format.

Number

For numbers, the input value indicates what is to be formatted. It is represented in standard C format, with three special values: Infinity, -Infinity, and NaN. If the application program does not format these values, then these input lines can be skipped.

Collation

The collation test area is special. Each result element consists of a number of lines, where each line compares as greater than or equal to the line before, according to the collation for the locale. The characters used come from the exemplar characters in CLDR, plus the set of tailored characters for the collation. In addition, the characters are closed under case; that means that if 'aa' is in the exemplar set, then all the combinations of case will show up: aa, aA, Aa, AA. That will sometimes result in some oddities: for example, 'ſ' long s will show up because it is a case variant of 's'.

On top of that, the strings are prefixed by 'x' or 'X', and sometimes postfixed by 'x'. This provides for a test of the strength of comparison. For example, when 'b' has a tertiary difference from 'B', the ordering will show up as:

XB
Xb
xB
xb
xBx
xbx

And in this locale (Danish), uppercase sorts first. Notice that every other line has a 'b' of a different case. With a secondary difference, we get the following type of pattern (for clarity, the uppercase variants are removed).

Xe
xe
Xé
xé
xex
xéx

Notice that rather than being every other line, as with the 'b's, the accented letters, having a secondary difference, are clumped together (except where a following 'x' separates them). With a primary difference, of course, the letters are completely separated (again, removing the uppercase characters for clarity):

Xb
xb
xbx
Xc
xc
xcx

If two strings are identical according to the collation (at the normal strength setting), then this is represented by repetition. For example, for Arabic we have:

xة
xت
xة<!-- dup for equality -->

This will guarantee the right ordering, since a testing program will have xة ≤ xت ≤ xة, which can only be true if  xة = xت. The data is presented simply as successive lines rather than multiple elements, so that multiple lines can simply be copied and used in other circumstances, without having to remove the element structure. (In addition, it makes the files much smaller.)
